# libx264偶数尺寸要求修复

## 🚨 新发现的问题

在修复了pad滤镜问题后，出现了新的FFmpeg错误：

```
[libx264 @ 0x130617ab0] height not divisible by 2 (854x641)
Error initializing output stream: Error while opening encoder for output stream #0:0
```

## 🔍 问题分析

### libx264编码器要求
**H.264编码器要求视频的宽度和高度都必须是偶数**

### 具体问题
- 原始视频：960×720
- 计算的480p质量：854×641
- 问题：641是奇数，libx264拒绝编码

### 技术原因
H.264编码使用宏块（macroblock），每个宏块是16×16像素。为了简化计算，libx264要求尺寸至少是2的倍数。

## ✅ 修复方案

### 1. 偶数尺寸强制转换
在质量配置计算后，强制将奇数尺寸转换为偶数：

```typescript
// 确保尺寸是偶数（libx264要求）
targetWidth = targetWidth % 2 === 0 ? targetWidth : targetWidth - 1;
targetHeight = targetHeight % 2 === 0 ? targetHeight : targetHeight - 1;
```

### 2. 修复位置
- `getAvailableQualities()` 方法
- 竖屏视频计算后
- 横屏视频计算后  
- 原始质量fallback后

## 🎯 修复效果

### 修复前（失败）：
- 960×720 → 480p：854×641 ❌（641是奇数）

### 修复后（成功）：
- 960×720 → 480p：854×640 ✅（640是偶数）
- 720×960 → 480p：360×480 ✅（都是偶数）
- 1280×720 → 360p：640×360 ✅（都是偶数）

## 📊 影响评估

### 质量影响
- **微乎其微**：减少1像素对视频质量几乎无影响
- **比例保持**：仍然保持原始视频比例
- **文件大小**：略微减小（像素更少）

### 兼容性
- **libx264**：完全兼容 ✅
- **其他编码器**：大多数也要求偶数尺寸 ✅
- **播放器**：所有现代播放器都支持 ✅

## 🔧 代码修复

1. **竖屏视频处理**：以高度为基准计算宽度后，强制偶数
2. **横屏视频处理**：以宽度为基准计算高度后，强制偶数
3. **原始质量fallback**：缩放后强制偶数

`src/video-processing/services/ffmpeg.service.ts` - `getAvailableQualities()` 方法
